# Phase 4.2 — Synthea data + FHIR MCP evidence retrieval

**Week**: 3 · **Status**: ⏳ Next

## Goal

End-to-end vertical slice: given a synthetic patient and a proposed service, retrieve and rank the clinical evidence package via FHIR MCP tools.

## Scope — what ships

1. **Synthea pipeline**
   - Module selection: MRI lumbar spine workflow (chronic low back pain + conservative therapy history) — the canonical PA test case.
   - Loaded into a local FHIR server (HAPI or Medplum sandbox; choice gated by ADR-0002).
   - ~50 synthetic patients with varied evidence-completeness, including deliberate "evidence missing" cases.

2. **FHIR MCP integration**
   - Reuse [`fhir-mcp-suite`](https://github.com/pcmedsinge/fhir-mcp-suite) as the MCP server.
   - Wire MCP tools into a minimal LangGraph `Evidence Gatherer` node.
   - Tools surfaced: `find_observations`, `find_conditions`, `find_procedures`, `find_medication_history`, `find_imaging_studies`, `find_documents`.

3. **Evidence ranking**
   - Deterministic first-pass: tag each retrieved resource against a hardcoded MRI-lumbar-spine policy checklist (e.g., "≥6 weeks conservative therapy", "neurological deficit", "failed PT").
   - LLM-mediated second pass: rank relevance and produce a one-line "why this matters" per resource.

4. **Demo**
   - One command: `make demo-evidence PATIENT=<id>` → prints structured evidence package to stdout + writes JSON.

## ADRs likely needed

- **ADR-0002** — FHIR server choice (HAPI vs Medplum vs Aidbox) for local dev.
- **ADR-0003** — How patient and service context is passed across the LangGraph state (typed TypedDict vs Pydantic model).

## Eval gates

| Metric | Target |
|---|---|
| Tool-call accuracy on 20-case golden set | ≥ 0.90 |
| Recall of "must-have" evidence resources (deterministic checklist) | 1.00 |
| Citation grounding (no fabricated resource references) | 1.00 |
| End-to-end latency (P50) | < 8s |
